# 設計方針（アーキテクチャノート）

このドキュメントは、機能拡張・保守を行う際に踏まえておくべき設計上の意図・トレードオフをまとめたもの。
「何を書いたか」はコードを読めばわかるので、ここには「なぜそうしたか」だけを書く。

## 責務分担: Go / フロント(JS)

**基本方針: バックエンド(Go)はブラウザのCORS/サンドボックス制約を突破するための汎用的なAPIだけを提供し、
機能・ロジックはすべてフロント(JS)側で実装する。** Go側にビジネスロジック（並び替え・カテゴリ分類など）を
持ち込まず、単純な窓口に徹する。

- UI描画・データ加工・並び替えロジックなど、ブラウザ側で完結する処理はすべてフロント(JS)側に置く。
- Go側（`App`, `app.go`）は、CORSやブラウザのサンドボックス制約でJSから直接到達できない処理のみを持つ。
  具体的には: 設定/データファイルの読み書き、外部アプリの起動（`OpenPath`）、
  ネイティブファイルドロップの受信、ファイル選択ダイアログ、アイコン抽出、テーマ検知。
- 新しい機能を追加する際も、まずフロント側で完結できないかを検討し、Go側の追加は
  「JSから到達できない処理」に限定する。

## データファイルの分割（`link-data.json` / `settings.json`）

リンク本体（`link-data.json`）とアプリ設定（`settings.json`）を分けて保存している。

- ウィンドウサイズ保存のような頻度の高い書き込みと、リンク一覧のような重要データの書き込みを分離することで、
  万一の書き込み失敗時の影響範囲を小さくする。
- 可読性の面でも、設定とデータが同一ファイルに混在するより見通しがよい。

## アセットハンドラ（HTML embed をしない理由 = 意図的なイレギュラー対応）

上記の基本方針（Go=汎用API、フロント=全機能）からすると、本来HTMLはGoバイナリに `embed` して
1個のexeとして完結させるのが素直な形。しかし本アプリでは意図的にそこから外れ、
`file_handler.go` の `fileHandler` が `frontend/link-launcher.html` をリクエストの都度
ディスクから読み込んで返す構成にしている（`embed` を使わない）。

- 理由: exe を再ビルドせずに HTML/CSS/JS を差し替えて挙動確認できるようにするため
  （開発中のイテレーションを優先した例外的な判断）。
- アクセス頻度が高いアプリではないため、キャッシュは実装していない（可読性優先）。

`wails.json` の `AssetServer.Assets` は常に `nil` にしている（`main.go`）。埋め込みindex.htmlが
Handlerより優先されてしまう問題を避けるため。

embedしていない分、配布時は `frontend/link-launcher.html` を別途同梱する必要がある。
パッケージング方法は [`DEV_NOTES.md`](./DEV_NOTES.md) の「配布パッケージング」を参照。

## ウィンドウサイズの優先順位

起動時のウィンドウサイズは次の優先順位で決定する:

```
起動引数(-width/-height) > 保存済み設定(settings.json) > デフォルト値(main.goのflagデフォルト)
```

`flag.Visit` で「引数が明示指定されたか」を判定している。`flag.Int` の戻り値だけでは
「未指定でデフォルト値と一致した場合」と区別できないため。

## テーマ検知はワンショット

`theme_windows.go` の `GetSystemTheme` は起動時に一度だけ Windows のレジストリ
（`HKCU\...\Personalize\AppsUseLightTheme`）を読む設計。OS設定の変更をリアルタイムに
追従はしない。追従させる場合は `RegNotifyChangeKeyValue` によるレジストリ変更通知が必要になるが、
本アプリのスコープでは過剰と判断し実装していない。

## ドラッグ&ドロップの二系統

- 外部（エクスプローラ等）からのファイル/フォルダのドロップ: Wails ネイティブの `OnFileDrop`
  （`app.go` の `onFileDrop`）で受信し、フルパスを取得する。
- アプリ内でのアイコン並び替え: HTML5 DnD（`draggable`属性）をフロント側で実装する想定。

ネイティブDnDとHTML5 DnDが同一WebView内でどう共存するか（座標系・発火順序）は
Wailsのバージョンに依存する可能性があり、実機での確認が必要な領域。

## アイコン抽出（`icon_windows.go`）

外部ライブラリに頼らず、GDI/Shell APIを直接 `syscall` 経由で呼び出す自作実装にしている
（`SHGetFileInfoW` → `GetIconInfo` → `GetDIBits` → PNGエンコード）。

- 追加の依存を増やさずに済む。
- 反面、DPI・アイコンサイズ・フォーマット周りの挙動は自前で保証する必要があり、
  実機での動作確認が前提の実装になっている。うまく動作しない場合は、
  自作実装を続けるか既存ライブラリへの差し替えを検討する（ライセンス・保守状況の確認が必要）。

## 外部パスを開く（`OpenPath`）

`cmd /c start` 経由でURL/ファイル/フォルダ/exeを既定アプリで開く。`start` はコマンド自体の
起動に成功すればエラーを返さないため、ローカルパス（`://` を含まないもの）は事前に
`os.Stat` で存在確認してから呼び出している。パスにスペースや特殊文字を含む場合の
エスケープは現状追加対応していない。

## 画像アイテム（種別"image"）とクリップボード画像貼り付け

`url`/`file`/`folder`/`exe`に加え、`.png/.jpg/.jpeg/.gif/.bmp/.webp/.ico`を`image`種別として扱う
（`app.go`の`ResolvePathType`）。クリックすると`OpenPath`でOSの既定ビューワを開くのではなく、
アプリ内蔵のコンパクトなビューワ（`#imgViewerOverlay`）で表示する。

- 画像の読み込みは常にGo側の`App.ReadImageDataURI(path)`を経由してBase64 data URIとして
  受け取る。`file://`をWebViewに直接読み込ませるとセキュリティ制約に引っかかる可能性があるため
  （責務分担の節で述べたCORS/サンドボックス回避の方針と同じ理由）。
- 画像ビューワは既存のフォーム系`#modalOverlay`を流用せず、独立した`#imgViewerOverlay`にした。
  流用すると「ビューワ表示用に一時的に書き換えた`#modal`のスタイルを、次にフォームモーダルを
  開く前に戻し忘れる」という状態管理上の事故が起きやすいため、最初から別要素にして
  リセット漏れの可能性自体を無くしている。

**クリップボードから貼り付けた画像（スクリーンショット等、CF_DIB）は`<exeDir>/images/`へ
別ファイル（PNG）として保存する方式を採用**（`link-data.json`へのdata URI埋め込みは、
件数が増えるとJSONが肥大化するため不採用。ユーザーに確認の上で決定）。保存後は他の
`file`種別アイテムと完全に同じ扱いにする。つまり**アプリからアイテムを削除してもファイル自体は
削除しない**（削除操作は「ランチャーへの登録解除」であって「実体の削除」ではない、という
このアプリ全体の一貫した扱い）。そのため`images/`フォルダ専用のクリーンアップ処理は持たせて
いない。不要なファイルは他の登録ファイルと同様、ユーザーが手動で削除する想定。
