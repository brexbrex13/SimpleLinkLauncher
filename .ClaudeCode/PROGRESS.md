# 実装進捗（GUI改善・差別化機能）

セッションをまたいで作業する際は、まずこのファイルを確認してから続きに着手すること。
運用ルール（レビュー無し・直接マージ可、リリース後にまとめて確認を取る）は `../CLAUDE.md` 参照。
各項目の実現性検討の結論は `DEV_NOTES.md` にも重複して書かず、ここに一本化する。

## 背景・方向性

Claunchとの差別化として「キーボード検索型ランチャーではなく、視覚的ダッシュボード/スピードダイヤル」の
方向性で合意済み（favicon対応を差別化の核として重視）。改善案の実現性は全項目「可能」と判断済み。
以下の順で段階的に実装する。

## フェーズ1: GUI改善一式 — 実装済み（実機未検証）

- [x] 1. ツールバー（コマンドバー＋タブ行）を1行に集約
- [x] 2. タイトルバー削除（`options.App.Frameless = true`、独自ドラッグ領域をCSSで指定）
- [x] 3. 最小化ボタン廃止、×（閉じる）ボタンのみ用意
- [x] 4. タイル⇔リスト切替ボタンを文字無し・アイコンのみに変更
- [x] 5. ウィンドウ位置も保存・復元（`Settings`に`WindowX`/`WindowY`追加、`WindowGetPosition`/`WindowSetPosition`活用）
- [x] 6. リンクフレーム（`#content`）外の領域でのスクロールでタブ切替
- [x] 7. リンクフレーム内スクロールの動的挙動
      （スクロールバー無し＝フレーム外と同挙動でタブ切替、スクロールバー有り＝端で少し尤度を持たせてタブ切替）
- [x] 8. 追加ボタン・検索窓を小さく
- [x] 9. カテゴリ設定の「…」ボタンを廃止し、カテゴリエリアの右クリックメニューに統合

検証方法: Windows実機・実際のWailsランタイムが無い環境のため、Chromiumのヘッドレスブラウザで
`frontend/link-launcher.html`を直接開き、レイアウト（1行ツールバー、複数タブでの折り返し無し）・
タイル/リスト切替・カテゴリ右クリックメニュー・ホイールでのタブ切替（フレーム外/フレーム内とも）の
動作をスクリーンショット付きで確認した。ただし以下はWails固有のランタイム機構に依存するため、
**実機(Windows)での確認が別途必要**（`DEV_NOTES.md`に記載済み）:
- `Frameless`によるタイトルバー削除の見た目、ドラッグ移動、自動リサイズ
- `runtime.Quit()`経由での終了とウィンドウ位置/サイズの保存
- `--wails-draggable`によるドラッグ領域とボタン類のクリック可否の両立

## フェーズ2: favicon取得 — 案A実装済み（実際のインターネットでの見え方は未検証）

- [x] 案A: `<img src="https://<domain>/favicon.ico">` を直接埋め込み（フロントのみ、Go側変更不要）。
      `iconWrapHtml()`で種別グリフの上に重ね、`onerror="this.remove()"`で失敗時はグリフに自動フォールバック。
      このサンドボックス環境は外部ドメインへの実アクセスができない（`curl`でも403）ため、
      ローカルHTTPサーバーを使い「favicon取得成功時」「失敗時のグリフフォールバック」の両方を
      Chromiumヘッドレスで動作確認済み。実際のWeb上のfaviconでの見た目（サイズ・画質・
      白背景での見え方）は次回リリース後にユーザー側で確認してほしい。
- [ ] 案Aで取得できないサイトが目立つ場合のみ: 案B `App.FetchFavicon(url)` をGoで追加し、
      `<head>`の`<link rel="icon">`を軽くパースして取得、Base64 data URIで返す（`ExtractIcon`と同パターン）

## フェーズ3: クリップボード貼り付け（空き枠右クリック→貼り付け）

- [ ] URL/テキストの貼り付け（Wailsランタイムの`ClipboardGetText()`を使用、既存`detectType()`へ接続）
- [ ] Explorerでコピーしたファイルオブジェクトの貼り付け
      （Win32クリップボードAPI: `OpenClipboard`→`GetClipboardData(CF_HDROP)`→`DragQueryFileW`を
      Goでsyscall実装。`icon_windows.go`と同系統のパターン。取得後は`onFileDrop`と同じ
      `ResolvePathType`→`ExtractIcon`の流れに接続）

## フェーズ4: 画像ビューワ・クリップボード画像対応

- [ ] 画像を項目として登録し、クリックでコンパクトなモーダルビューワ表示
      （既存`#modalOverlay`パターンを流用。画像読み込みは`file://`直読みを避け、
      `App.ReadImageAsDataURI(path)`をGoで追加してBase64で渡す）
- [ ] クリップボードから直接画像を貼り付け（一時保管・クリップボードマネージャ的用途）
      （CF_DIBの読み取りをGoで実装。`icon_windows.go`の`GetDIBits`まわりのロジックを流用可能。
      **要設計判断（リリース後にユーザーへ確認）**: 画像データを別ファイルとして保存するか
      `link-data.json`にdata URIで埋め込むか。埋め込み案は実装が単純だが件数が増えるとJSONが肥大化する。
      "一時的な保管"という性質上の破棄/クリーンアップ方針も未確定）
