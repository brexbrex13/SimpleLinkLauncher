# 実装進捗（GUI改善・差別化機能）

セッションをまたいで作業する際は、まずこのファイルを確認してから続きに着手すること。
運用ルール（レビュー無し・直接マージ可、リリース後にまとめて確認を取る）は `../CLAUDE.md` 参照。
各項目の実現性検討の結論は `DEV_NOTES.md` にも重複して書かず、ここに一本化する。

## 背景・方向性

Claunchとの差別化として「キーボード検索型ランチャーではなく、視覚的ダッシュボード/スピードダイヤル」の
方向性で合意済み（favicon対応を差別化の核として重視）。改善案の実現性は全項目「可能」と判断済み。
以下の順で段階的に実装する。

## フェーズ1: GUI改善一式 — 実装済み（実機未検証）

- [x] 1. ツールバー（コマンドバー＋タブ行）を1行に集約
- [x] 2. タイトルバー削除（`options.App.Frameless = true`、独自ドラッグ領域をCSSで指定）
- [x] 3. 最小化ボタン廃止、×（閉じる）ボタンのみ用意
- [x] 4. タイル⇔リスト切替ボタンを文字無し・アイコンのみに変更
- [x] 5. ウィンドウ位置も保存・復元（`Settings`に`WindowX`/`WindowY`追加、`WindowGetPosition`/`WindowSetPosition`活用）
- [x] 6. リンクフレーム（`#content`）外の領域でのスクロールでタブ切替
- [x] 7. リンクフレーム内スクロールの動的挙動
      （スクロールバー無し＝フレーム外と同挙動でタブ切替、スクロールバー有り＝端で少し尤度を持たせてタブ切替）
- [x] 8. 追加ボタン・検索窓を小さく
- [x] 9. カテゴリ設定の「…」ボタンを廃止し、カテゴリエリアの右クリックメニューに統合

検証方法: Windows実機・実際のWailsランタイムが無い環境のため、Chromiumのヘッドレスブラウザで
`frontend/link-launcher.html`を直接開き、レイアウト（1行ツールバー、複数タブでの折り返し無し）・
タイル/リスト切替・カテゴリ右クリックメニュー・ホイールでのタブ切替（フレーム外/フレーム内とも）の
動作をスクリーンショット付きで確認した。ただし以下はWails固有のランタイム機構に依存するため、
**実機(Windows)での確認が別途必要**（`DEV_NOTES.md`に記載済み）:
- `Frameless`によるタイトルバー削除の見た目、ドラッグ移動、自動リサイズ
- `runtime.Quit()`経由での終了とウィンドウ位置/サイズの保存
- `--wails-draggable`によるドラッグ領域とボタン類のクリック可否の両立

## フェーズ2: favicon取得 — 案A実装済み（実際のインターネットでの見え方は未検証）

- [x] 案A: `<img src="https://<domain>/favicon.ico">` を直接埋め込み（フロントのみ、Go側変更不要）。
      `iconWrapHtml()`で種別グリフの上に重ね、`onerror="this.remove()"`で失敗時はグリフに自動フォールバック。
      このサンドボックス環境は外部ドメインへの実アクセスができない（`curl`でも403）ため、
      ローカルHTTPサーバーを使い「favicon取得成功時」「失敗時のグリフフォールバック」の両方を
      Chromiumヘッドレスで動作確認済み。実際のWeb上のfaviconでの見た目（サイズ・画質・
      白背景での見え方）は次回リリース後にユーザー側で確認してほしい。
- [ ] 案Aで取得できないサイトが目立つ場合のみ: 案B `App.FetchFavicon(url)` をGoで追加し、
      `<head>`の`<link rel="icon">`を軽くパースして取得、Base64 data URIで返す（`ExtractIcon`と同パターン）

## フェーズ3: クリップボード貼り付け（空き枠右クリック→貼り付け）

- [x] URL/テキストの貼り付け（Wailsランタイムの`ClipboardGetText()`を使用）。空き枠の右クリックメニューに
      「📋 クリップボードから貼り付け」を追加。クリップボードのテキストを追加モーダルにプリフィルする形にした
      （即時追加ではなくモーダル経由。名前確認・カテゴリ選択を既存フローと共通化するため）。
      種別判定は保存時の既存`detectType()`に任せているため、URLはそのままURL種別になる。
      ヘッドレスブラウザで`window.runtime.ClipboardGetText`をモックし、右クリック→貼り付け→
      モーダルへのプリフィルまで動作確認済み。
- [x] Explorerでコピーしたファイルオブジェクトの貼り付け（実機未検証）。
      新規`clipboard_windows.go`に`App.PasteClipboardFiles() ([]DroppedItem, error)`を追加。
      `OpenClipboard`→`GetClipboardData(CF_HDROP)`→`DragQueryFileW`をsyscallで実装
      （`icon_windows.go`と同系統のパターン）。パス解決は`onFileDrop`と共通化した
      `buildDroppedItems()`（`app.go`）を経由するため`ResolvePathType`の判定ロジックは重複していない。
      フロント側`pasteFromClipboard()`は「①ファイルオブジェクトを試す→無ければ②テキストとして
      モーダルにプリフィル」の順で処理する。①の挿入は同カテゴリの「空き枠を埋める」操作と
      挙動を揃え、1件目はクリックした空き枠自体を置き換え、2件目以降は直後に新規スロットとして
      挿入する（複数ファイルを一括コピーした場合）。
      `frontend/wailsjs/go/main/App.{js,d.ts}`は`wails`CLIが無い環境のため手動で追記した
      （次回実機で`wails build`する際に整合性を確認すること）。
      ヘッドレスブラウザで`window.go.main.App.PasteClipboardFiles`をモックし、
      ファイル/フォルダの挿入と、空配列を返した場合のテキストへのフォールバックの両方を確認済み。
      実機でのクリップボードAPI自体の動作（`OpenClipboard`失敗時の挙動含む）は未検証
      （`.ClaudeCode/DEV_NOTES.md`参照）。

## フェーズ4: 画像ビューワ・クリップボード画像対応

- [x] 画像を項目として登録し、クリックでコンパクトなモーダルビューワ表示（実機未検証）。
      新しい種別"image"を追加（`app.go`の`ResolvePathType`が`.png/.jpg/.jpeg/.gif/.bmp/.webp/.ico`を
      判定、`App.ReadImageDataURI(path)`がファイルを読みBase64 data URIを返す）。
      フロント側は`GLYPHS/TYPE_COLOR/TYPE_LABEL`に`image`を追加し、バッジ色（`--badgeImageFg/Bg`、
      紫系）も両テーマに追加。既存のフォーム系`#modalOverlay`は流用せず、独立した
      `#imgViewerOverlay`（背景クリック/×ボタンで閉じる、`isOverlayOpen()`にも追加済みなので
      表示中はホイールタブ切替が誤発火しない）を新設した。理由: フォームモーダルの
      スタイル（幅・パディング等）をビューワ用に一時的に書き換えて戻し忘れる、という
      状態管理上の事故を避けるため。
      画像アイテムのアイコンは`ExtractIcon`(OSのシェルアイコン)ではなく`fetchImageThumbnail()`
      経由で実際の画像内容をサムネイルにしている（D&D・クリップボード貼り付け・複数ファイル
      ブラウズ・単体追加のすべての登録経路で呼ぶように統一済み）。
      ヘッドレスブラウザで、タイルクリックでのビューワ表示・閉じるボタン・ビューワ表示中の
      ホイール操作でタブが切り替わらないこと・貼り付け登録後のサムネイル反映を確認済み。
      `frontend/wailsjs/go/main/App.{js,d.ts}`にも`ReadImageDataURI`を手動追記済み。
- [x] クリップボードから直接画像を貼り付け（スクリーンショット等、実機未検証）。
      ユーザーに確認の上、**保存方式は「別ファイルとして保存」を採用**（`link-data.json`への
      data URI埋め込み案は不採用）。`App.PasteClipboardImage()`（`clipboard_windows.go`）が
      CF_DIB形式（24bit/32bit BI_RGBのみ対応）をGDI無しで直接メモリ解析してPNGにエンコードし、
      `<exeDir>/images/`へ`clip_<タイムスタンプ>.png`として保存、そのパスを返す。
      保存後は通常の画像アイテム（上記）と全く同じ扱いにする（＝アプリからアイテムを削除しても
      ファイル自体は消さない。他の"file"種別アイテムと同じ「登録解除であって実体削除ではない」
      という既存方針にそのまま乗せているだけなので、専用のクリーンアップ処理は追加していない。
      不要になったファイルは`images/`フォルダから手動で削除する想定）。
      空き枠の「📋 クリップボードから貼り付け」は優先順位を
      「①ファイルオブジェクト(CF_HDROP) → ②生の画像データ(CF_DIB、今回追加) → ③テキスト」の順に
      試すよう拡張した。
      ヘッドレスブラウザで`PasteClipboardImage`をモックし、画像アイテムとして登録されサムネイルが
      反映されることを確認済み。

## フェーズ5: v0.1.0実機フィードバックの修正

v0.1.0を実機検証したユーザーからのフィードバック。各項目は`TaskCreate`でも管理しているので
セッション再開時は`TaskList`も確認すること。

- [ ] 1. タブオーバーフロー時、ホイールでタブ切替してもアクティブタブがスクロール範囲外で
      見えないことがある。タブ切替時にアクティブタブを`scrollIntoView({inline:'nearest'})`する。
- [ ] 2. クリップボードのテキスト/URL貼り付けが現状「追加モーダルを開いてプリフィル」に
      なっているが、**即時登録**に変更する（編集したい場合は登録後に右クリック→編集）。
      URL種別の場合はGo側新規メソッド`FetchPageTitle(url)`でHTMLの`<title>`タグを取得し、
      項目名として使う（取得失敗時はパス由来の名前にフォールバック）。
- [ ] 3. クリップボードに画像データ（スクリーンショット等、CF_DIB/CF_DIBV5）をコピーした状態での
      貼り付けが失敗する（ファイルオブジェクトとしてのコピー＆貼り付けは成功する）。
      `PasteClipboardImage`が`biCompression != 0`（BI_RGB以外）を一律拒否しているのが原因の
      可能性が高い。多くのスクリーンショットツールは32bitで`BI_BITFIELDS`（値3）を使うため、
      これも許容し、かつ「古い形式のBITMAPINFOHEADER(biSize=40)+外部カラーマスク12バイト」
      パターンのオフセット計算も対応する。実機未検証のため引き続き注意（`DEV_NOTES.md`参照）。
- [ ] 4. 画像ビューワにホイールでの拡大縮小を追加する（`transform:scale()`、開くたびに倍率リセット）。
      ※Wails v2はマルチウィンドウ（サブウィンドウ）に対応していないことを確認済み
      （vendored `wailsapp/wails/v2` にウィンドウ生成APIが存在しない。v2は1アプリ1ウィンドウが前提で、
      複数ウィンドウ対応はv3以降の機能）。そのため画像ビューワは今後もページ内オーバーレイとして
      実装する方針で変わらない。
- [ ] 5. タイル表示の追加(+)ボタン（`addSlotTile`）にも右クリックメニュー（空き枠相当）を許可する。
- [ ] 6. リスト表示の末尾にある「謎のブランクパネル」（ドロップ受け皿だけのdiv）を、
      左クリック=追加モーダル、右クリック=空き枠メニュー、の見える+ボタンとして実装し直す。
      これによりリスト表示でもクリップボード貼り付けが可能になる。
- [ ] 7. フレームレスウィンドウの右端リサイズが効かない（左端は可能）。`#content`の縦スクロールバーが
      ウィンドウ右端ぴったりに重なり、Wailsの端リサイズ判定（6px）と衝突している可能性が高い。
      `#content`に数px右マージンを設けて回避を試みる（実機未検証、次回リリースで要確認）。
- [ ] 8. タブ・カテゴリの件数バッジを削除してシンプル表示にする。
- [ ] 9. タイルのアイコン表示が小さく感じられる。タイルの枠（背景または境界線）に種別カラーを
      持たせ、タイル内は「アイコン＋タイトル（1行、はみ出しは…で省略）」のみのシンプルな構成にする。
      アイコンサイズも拡大する。list/search表示は現状維持。
- [ ] 10. favicon取得がタブ切替・再描画のたびに重複して発生している（`iconWrapHtml()`が
      毎レンダリングで新しい`<img>`を生成し直しているため）。画像サムネイル/ExtractIconと同じ
      「一度だけ取得してitem.iconへ永続化」パターンに統一する。`iconWrapHtml()`は副作用の無い
      純粋な描画関数に戻し、favicon取得はアイテム新規作成時＋`init()`時の既存アイテム一括
      スキャンの2箇所でのみ1回実行する。

## 保留中の設計判断（ユーザー確認待ち）

- **グローバルの「＋追加」ボタン（ツールバー左端）の要否**: ユーザーからは「タブの＋ボタンと
  紛らわしい」「空きパネル→右クリックで登録、を基本にした方がいいのでは」という指摘あり。
  ただし「少し直感的ではなくなるが」とも書かれており、ユーザー自身も迷っている様子。
  空きパネルでの右クリック登録（上記5・6で強化済み）が確立した後、グローバル＋ボタンを
  廃止するかどうかは次回のフィードバックで確認してから判断する。

## 新機能案（規模が大きいため実装前に方向性を確認する）

- **お気に入り／よく使うリンクランキング**: ホームタブを本当の意味での「ホーム」にし、
  「お気に入り」（項目ホバー時に右上に★表示、クリックでトグル）と「よく開くファイルの
  ランキング（上位10件程度）」のカテゴリを設ける案。ホームタブのアイコンを家マークにし、
  ＋ボタンの隣に専用の家ボタンを置く案も。要件（お気に入りの永続化先、開いた回数の
  カウント方法、ランキングの集計範囲・更新タイミング等）をユーザーと詰めてから着手する。
- **カテゴリの開閉（アコーディオン）**: デフォルトはオープン、可能なら開閉状態を記憶する。
  比較的小規模な機能なので、優先度が上がれば単独で着手しやすい。
- **フォルダのツリー表示ポップアップ**: フォルダ登録項目を開く際、Explorerで開く代わりに
  ポップアップでツリー表示できないか、という提案。実現するなら
  Go側に`ListDirectory(path) ([]FileEntry, error)`（直下の子要素一覧を返す）を追加し、
  フロント側で遅延展開するツリーUIを実装する形になる。さらに「項目ごとにExplorer起動/
  ポップアップ表示を選べる設定」も追加する案。規模が大きいため、まず「本当に必要か」
  「Explorerで十分ではないか」をユーザーに確認してから着手するかどうかを決める。
