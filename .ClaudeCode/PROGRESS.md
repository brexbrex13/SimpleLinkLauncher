# 実装進捗（GUI改善・差別化機能）

セッションをまたいで作業する際は、まずこのファイルを確認してから続きに着手すること。
運用ルール（レビュー無し・直接マージ可、リリース後にまとめて確認を取る）は `../CLAUDE.md` 参照。
各項目の実現性検討の結論は `DEV_NOTES.md` にも重複して書かず、ここに一本化する。

## 背景・方向性

Claunchとの差別化として「キーボード検索型ランチャーではなく、視覚的ダッシュボード/スピードダイヤル」の
方向性で合意済み（favicon対応を差別化の核として重視）。改善案の実現性は全項目「可能」と判断済み。
以下の順で段階的に実装する。

- `--wails-draggable`によるドラッグ領域とボタン類のクリック可否の両立

## フェーズ2: favicon取得 — 案A実装済み（実際のインターネットでの見え方は未検証）

- [x] 案A: `<img src="https://<domain>/favicon.ico">` を直接埋め込み（フロントのみ、Go側変更不要）。
      `iconWrapHtml()`で種別グリフの上に重ね、`onerror="this.remove()"`で失敗時はグリフに自動フォールバック。
      このサンドボックス環境は外部ドメインへの実アクセスができない（`curl`でも403）ため、
      ローカルHTTPサーバーを使い「favicon取得成功時」「失敗時のグリフフォールバック」の両方を
      Chromiumヘッドレスで動作確認済み。実際のWeb上のfaviconでの見た目（サイズ・画質・
      白背景での見え方）は次回リリース後にユーザー側で確認してほしい。
- [ ] 案Aで取得できないサイトが目立つ場合のみ: 案B `App.FetchFavicon(url)` をGoで追加し、
      `<head>`の`<link rel="icon">`を軽くパースして取得、Base64 data URIで返す（`ExtractIcon`と同パターン）

## フェーズ3: クリップボード貼り付け（空き枠右クリック→貼り付け）

- [x] URL/テキストの貼り付け（Wailsランタイムの`ClipboardGetText()`を使用）。空き枠の右クリックメニューに
      「📋 クリップボードから貼り付け」を追加。クリップボードのテキストを追加モーダルにプリフィルする形にした
      （即時追加ではなくモーダル経由。名前確認・カテゴリ選択を既存フローと共通化するため）。
      種別判定は保存時の既存`detectType()`に任せているため、URLはそのままURL種別になる。
      ヘッドレスブラウザで`window.runtime.ClipboardGetText`をモックし、右クリック→貼り付け→
      モーダルへのプリフィルまで動作確認済み。
- [x] Explorerでコピーしたファイルオブジェクトの貼り付け（実機未検証）。
      新規`clipboard_windows.go`に`App.PasteClipboardFiles() ([]DroppedItem, error)`を追加。
      `OpenClipboard`→`GetClipboardData(CF_HDROP)`→`DragQueryFileW`をsyscallで実装
      （`icon_windows.go`と同系統のパターン）。パス解決は`onFileDrop`と共通化した
      `buildDroppedItems()`（`app.go`）を経由するため`ResolvePathType`の判定ロジックは重複していない。
      フロント側`pasteFromClipboard()`は「①ファイルオブジェクトを試す→無ければ②テキストとして
      モーダルにプリフィル」の順で処理する。①の挿入は同カテゴリの「空き枠を埋める」操作と
      挙動を揃え、1件目はクリックした空き枠自体を置き換え、2件目以降は直後に新規スロットとして
      挿入する（複数ファイルを一括コピーした場合）。
      `frontend/wailsjs/go/main/App.{js,d.ts}`は`wails`CLIが無い環境のため手動で追記した
      （次回実機で`wails build`する際に整合性を確認すること）。
      ヘッドレスブラウザで`window.go.main.App.PasteClipboardFiles`をモックし、
      ファイル/フォルダの挿入と、空配列を返した場合のテキストへのフォールバックの両方を確認済み。
      実機でのクリップボードAPI自体の動作（`OpenClipboard`失敗時の挙動含む）は未検証
      （`.ClaudeCode/DEV_NOTES.md`参照）。

## フェーズ4: 画像ビューワ・クリップボード画像対応

- [x] 画像を項目として登録し、クリックでコンパクトなモーダルビューワ表示（実機未検証）。
      新しい種別"image"を追加（`app.go`の`ResolvePathType`が`.png/.jpg/.jpeg/.gif/.bmp/.webp/.ico`を
      判定、`App.ReadImageDataURI(path)`がファイルを読みBase64 data URIを返す）。
      フロント側は`GLYPHS/TYPE_COLOR/TYPE_LABEL`に`image`を追加し、バッジ色（`--badgeImageFg/Bg`、
      紫系）も両テーマに追加。既存のフォーム系`#modalOverlay`は流用せず、独立した
      `#imgViewerOverlay`（背景クリック/×ボタンで閉じる、`isOverlayOpen()`にも追加済みなので
      表示中はホイールタブ切替が誤発火しない）を新設した。理由: フォームモーダルの
      スタイル（幅・パディング等）をビューワ用に一時的に書き換えて戻し忘れる、という
      状態管理上の事故を避けるため。
      画像アイテムのアイコンは`ExtractIcon`(OSのシェルアイコン)ではなく`fetchImageThumbnail()`
      経由で実際の画像内容をサムネイルにしている（D&D・クリップボード貼り付け・複数ファイル
      ブラウズ・単体追加のすべての登録経路で呼ぶように統一済み）。
      ヘッドレスブラウザで、タイルクリックでのビューワ表示・閉じるボタン・ビューワ表示中の
      ホイール操作でタブが切り替わらないこと・貼り付け登録後のサムネイル反映を確認済み。
      `frontend/wailsjs/go/main/App.{js,d.ts}`にも`ReadImageDataURI`を手動追記済み。
- [x] クリップボードから直接画像を貼り付け（スクリーンショット等、実機未検証）。
      ユーザーに確認の上、**保存方式は「別ファイルとして保存」を採用**（`link-data.json`への
      data URI埋め込み案は不採用）。`App.PasteClipboardImage()`（`clipboard_windows.go`）が
      CF_DIB形式（24bit/32bit BI_RGBのみ対応）をGDI無しで直接メモリ解析してPNGにエンコードし、
      `<exeDir>/images/`へ`clip_<タイムスタンプ>.png`として保存、そのパスを返す。
      保存後は通常の画像アイテム（上記）と全く同じ扱いにする（＝アプリからアイテムを削除しても
      ファイル自体は消さない。他の"file"種別アイテムと同じ「登録解除であって実体削除ではない」
      という既存方針にそのまま乗せているだけなので、専用のクリーンアップ処理は追加していない。
      不要になったファイルは`images/`フォルダから手動で削除する想定）。
      空き枠の「📋 クリップボードから貼り付け」は優先順位を
      「①ファイルオブジェクト(CF_HDROP) → ②生の画像データ(CF_DIB、今回追加) → ③テキスト」の順に
      試すよう拡張した。
      ヘッドレスブラウザで`PasteClipboardImage`をモックし、画像アイテムとして登録されサムネイルが
      反映されることを確認済み。

## フェーズ5: v0.1.0実機フィードバックの修正

v0.1.0を実機検証したユーザーからのフィードバック。各項目は`TaskCreate`でも管理しているので
セッション再開時は`TaskList`も確認すること。

- [x] 1. タブオーバーフロー時、ホイールでタブ切替してもアクティブタブがスクロール範囲外で
      見えないことがある。`renderTabs()`でアクティブタブを`scrollIntoView({inline:'nearest',block:'nearest'})`
      するようにした。
- [x] 2. クリップボードのテキスト/URL貼り付けを「追加モーダルを開いてプリフィル」から
      **即時登録**に変更した（編集したい場合は登録後に右クリック→編集）。Go側新規メソッド
      `FetchPageTitle(url)`（`net/http`でGET、先頭64KBのみ読み取り`<title>`タグを正規表現で
      抽出、`html.UnescapeString`＋空白正規化）を追加し、URL種別の場合はページタイトルを
      項目名として使う（取得失敗時はURL文字列そのものにフォールバック）。ヘッドレスChromium
      で「URL+取得成功」「URL+取得失敗→フォールバック」「非URLテキスト→即時登録・モーダル
      非表示」の3パターンを確認済み。
- [x] 3. クリップボード画像（CF_DIB）貼り付け失敗の原因だった`biCompression != 0`一律拒否を
      修正。`BI_BITFIELDS`（値3、16/32bit）を許容し、R/G/Bの3個のカラーマスク（クラシックな

      読み取ってピクセルごとにマスクベースで色成分を復元する`maskToU8()`を追加した。
      オフセット計算・マスク復元ロジックは独立したGoスクリプトで数値シミュレーションして
      正しさを確認済み（実機のWin32 syscall自体は本セッションでは検証不可のため、
      実機でのSnipping Tool等での動作確認は引き続き必要。`.ClaudeCode/DEV_NOTES.md`参照）。
- [x] 4. 画像ビューワにホイールでの拡大縮小を追加した（`#imgViewerBox`のwheelイベントで

      ※Wails v2はマルチウィンドウ（サブウィンドウ）に対応していないことを確認済み

      複数ウィンドウ対応はv3以降の機能）。そのため画像ビューワは今後もページ内オーバーレイとして
      実装する方針で変わらない。
- [x] 5. タイル表示の追加(+)ボタン（`addSlotTile`）にも右クリックメニュー（空き枠相当）を許可した。
      末尾位置（配列末尾より先の仮想位置）では「この空き枠を削除」の項目は意味が無いため非表示にする
      `isTail`引数を`openEmptySlotContextMenu()`に追加した。
- [x] 6. リスト表示の末尾にある「謎のブランクパネル」（ドロップ受け皿だけのdiv）を、

- [x] 7. フレームレスウィンドウの右端リサイズが効かない（左端は可能）という報告への対応として、

      重ならないようにした（Wailsの端リサイズ判定6pxとの衝突を避ける狙い。**実機未検証の仮説に
      基づく対応**、次回リリースで実際に直ったか確認が必要）。
- [x] 8. タブ・カテゴリの件数バッジを削除し、シンプル表示にした。
- [x] 9. タイルのアイコン表示が小さく感じられる問題への対応として、タイル左枠を種別カラーで

      アイテム新規作成時（`insertItemsAt()`・`openAddModal()`）と`init()`時の既存アイテム

## フェーズ6: ツールバー2行化・グローバル＋ボタンの用途確定（実装済み・実機未検証）

v0.2.0リリース後、ユーザーから「新機能案・保留中の設計判断は全部やりたい」との方向性確認を得た。
着手順は「グローバル＋ボタンの要否 → カテゴリアコーディオン → お気に入り/ランキング →
フォルダツリー表示」の順で合意（まとめてできるなら一気通貫でも可、そうでなければ上から）。

グローバル＋ボタンの要否を検討する過程で、リスト表示時にタブ数が増えると窮屈になる問題が
浮上し、ツールバーを2行に分ける方向で合意。その結果グローバル＋ボタンは「廃止」ではなく
「リンク追加ボタンとして残す」方向に転換した（経緯: タブ管理系の＋を右クリックメニューへ
移したことで、当初の廃止理由だった「タブの＋ボタンと紛らわしい」という問題自体が解消された
ため）。

- [x] ツールバーを2行に分割（`frontend/link-launcher.html`）。
      1行目(`#toolbar`)=アプリ全体系（検索窓・タイル/リスト切替・🌓・✕・左端に「＋追加」）、
      2行目(`#tabBar`)=タブストリップ専用（全幅）。両行とも`--wails-draggable:drag`
      （ボタン/inputのみ`no-drag`）でドラッグ領域を維持。
- [x] タブストリップの見た目をモダン化: スクロールバーを非表示にし(`scrollbar-width:none`+
      `::-webkit-scrollbar{display:none}`)、`bindTabStripDragScroll()`でマウスドラッグに
      よる横スクロールを実装。スクロールスナップは付けていないため、端で次タブが半分だけ
      見える「peek」表示になる（ヘッドレスChromiumで実タブ12個を用意しドラッグ→
      `scrollLeft`の変化とドラッグ後の誤クリック抑止を確認済み）。ホイールでのタブ切替は
      `#app`レベルの既存リスナー(`bindWheelTabSwitch`)のままで、行分割の影響を受けない。
- [x] タブ追加を専用ボタン(`btnAddTab`)から、既存タブの右クリックメニュー
      (`openTabContextMenu`)内の「＋ 新しいタブを追加」に統合。タブが0件になることは
      削除時のガード（最後の1つは削除不可）により発生しないため、右クリック導線のみで
      問題ない。
- [x] 「＋追加」モーダル(`openAddModal`)に「タブ」選択を追加（既存の移動/複製モーダル
      `openMoveCopyModal`と同じ「タブ選択→カテゴリ選択肢を動的に再構築」パターンを流用）。
      編集・空き枠埋め・純粋な新規追加の3パターンすべてで、追加/移動先タブをこの場で
      選べるようになった（保存時、選択タブが現在のアクティブタブと異なればそちらへ切り替える）。
      ヘッドレスChromiumで、タブ切替に応じたカテゴリ選択肢の再描画、右クリックメニューの
      表示内容を確認済み。実機(Windows/Wailsランタイム)での確認は引き続き必要。

## フェーズ8: ホームタブ・お気に入り・よく使うランキング

ユーザーとの合意事項:
- お気に入り／ランキングはホームタブの専用集約ビューにまとめる（実体は元のタブ/カテゴリの
  ままで、フラグ/カウントを立てるだけ）。ホームタブへの個別リンク登録は不可。
- 表示順は「★お気に入り」→「よく使う」。
- 「よく使う」は累計オープン回数（`item.openCount`）でソートし上位10件。
- ホームタブはタブストリップの先頭に家アイコン（テキストラベル無し）で表示する。

- [x] `HOME_TAB_ID`固定IDの特別なホームタブを新設（`isHome:true`、`categories`は常に空配列）。
      `defaultData()`は新規インストール時にホーム＋通常タブ「メイン」の2つを作る。
      既存インストール（`isHome`タブが無いデータ）は`init()`でホームタブを先頭に追加する
      マイグレーションを実装。**既存のタブ（たまたま「ホーム」という名前だったものを含む）は
      一切変更しない**（実データの誤った上書き/移行を避けるため、安全側に倒した設計）。
      ヘッドレスChromiumで、旧データ（`isHome`無し、実名「ホーム」タブに実データあり）を
      読み込ませ、新ホームタブが追加されつつ旧タブが無傷で残ることを確認済み。
- [x] タブストリップでホームタブのみ家アイコン(SVG)を表示（`renderTabs()`の`tab.isHome`分岐）。
      右クリックメニューも専用化し、「＋ 新しいタブを追加」のみ表示（名前変更/カテゴリ追加/
      削除は不可能・意味を持たないため非表示）。
- [x] タブ追加ボタン(`btnAddTab`)をタブストリップ末尾に復活（フェーズ6で右クリックのみに
      していたが、既存の「カテゴリ追加」も常設ボタン＋右クリックの両方を持つ前例に揃えた）。
- [x] アイテム右クリックメニューに「☆ お気に入りに追加/★ お気に入りを解除」を追加
      (`item.favorite`をトグルし永続化)。タイル/リストにも★バッジを表示(`.favStar`)。
- [x] `handleOpen(item)`で開くたびに`item.openCount`をインクリメントして永続化。
